# 웹 해킹 입문 · OWASP Juice Shop

웹사이트는 어떻게 뚫릴까? 이번 시간엔 **일부러 허술하게 만든 쇼핑몰**을 직접 공격해보면서,
실제 해커들이 가장 많이 쓰는 수법의 원리를 손으로 익힙니다. 모든 데이터는 가짜라서 마음껏 부숴도 됩니다.

> **Juice Shop**은 전 세계 보안 교육·대회에서 쓰는 표준 실습용 웹앱입니다.
> 실제 서비스가 아니라, "여기 취약점이 숨어 있으니 찾아봐"라고 만들어 둔 연습장이에요.

## 오늘의 목표

- 웹앱에 입력한 값이 서버에서 어떻게 처리되는지 감을 잡는다
- **SQL 인젝션**으로 비밀번호 없이 로그인을 뚫어본다
- **XSS**로 남의 화면에서 내 코드가 실행되게 해본다
- 이 모든 게 결국 "입력값을 그대로 믿어서" 생긴 문제임을 이해한다

## 진행 방식

학생들이 **각자 노트북에서 직접** 합니다. 강사는 시작/접속만 안내하고, 막히는 지점에서 힌트를 줍니다.
- 강사 PC: `http://localhost:3000`
- 학생: 화면에 표시된 `학생 접속` 주소로 접속 (같은 와이파이)

---

## 단계 1 · 정찰하기 (10분)

해킹의 시작은 공격이 아니라 **관찰**입니다.

1. 사이트를 둘러보고, 상품을 장바구니에 담고, 회원가입을 해봅니다.
2. <kbd>F12</kbd>로 개발자 도구를 열고 **Network 탭**을 봅니다. 버튼을 누를 때마다 브라우저가 서버에 무슨 요청을 보내는지 관찰하세요.
3. 스스로 질문해 봅니다: *"내가 입력한 값은 어디로 갈까? 누가, 어떻게 처리할까?"*

> 공격자는 화면에 보이는 것보다 **그 뒤에서 오가는 요청**을 봅니다. 그 시야를 갖는 게 첫걸음이에요.

## 단계 2 · SQL 인젝션으로 로그인 뚫기 (20분)

로그인 창의 이메일 칸에 아래를 입력하고, 비밀번호는 아무거나 칩니다.

```
' OR 1=1 --
```

비밀번호를 모르는데 로그인이 됩니다. 왜 그럴까요?

- 서버는 보통 이런 식으로 쿼리를 만듭니다: `SELECT * FROM users WHERE email='입력값' AND password='입력값'`
- 입력값이 검사 없이 그대로 끼어들면, `OR 1=1`(항상 참) 때문에 **조건이 무력화**됩니다.
- 즉, 입력값이 데이터가 아니라 **명령어**로 해석된 것 — 이게 인젝션입니다.

**막는 법:** Prepared Statement(파라미터 바인딩)로 입력값을 '명령어'가 아닌 '데이터'로만 다루기, 그리고 입력값 검증.

## 단계 3 · XSS 맛보기 (15분)

검색창이나 입력 폼에 다음을 넣어봅니다.

```html
<iframe src="javascript:alert('XSS')">
```

경고창이 뜨면, 내가 넣은 코드가 **그 페이지를 보는 사람의 브라우저에서 실행**된 겁니다.

- 지금은 경고창이지만, 같은 방법으로 쿠키를 훔치거나 화면을 조작할 수 있습니다. (다음 회차에서 직접 해봅니다.)
- 원인은 똑같습니다: 사용자 입력을 **그대로 화면에 출력**했기 때문.

**막는 법:** 출력 인코딩(이스케이프), Content-Security-Policy.

## 정리 토론 (15분)

- 오늘 본 두 취약점의 공통 원인은 무엇이었나요? → **"입력값을 믿었다."**
- 개발자라면 어디서 막을 수 있었을까요?
- 더 도전하고 싶다면: Juice Shop 우측 상단 `Score Board`(`#/score-board`)에 난이도별 과제가 있습니다.

## 꼭 기억하기

여기서 배운 기술을 **허락 없이 실제 사이트에 시도하면 범죄**입니다(정보통신망법 등).
우리는 "어떻게 뚫리는지"를 알아야 "어떻게 막는지"를 알 수 있어서 공격을 배우는 것입니다.

> **강사 메모.** 데이터가 꼬이면 **종료 → 시작**으로 깨끗한 상태가 됩니다. 학생들이 동시에 접속해도 각자 세션이라 충돌하지 않습니다.
