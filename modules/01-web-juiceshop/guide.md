# 웹 해킹 입문 · OWASP Juice Shop

> OWASP Juice Shop은 보안 교육을 위해 **일부러 취약하게** 만든 오픈소스 쇼핑몰 웹앱입니다.
> 실제 서비스가 아니며, 모든 데이터는 가짜입니다. 마음껏 공격해도 안전합니다.

## 0. 시작하기 (강사)

1. 위 **[▶ 시작]** 버튼을 누른다. (최초 1회는 이미지 다운로드로 1~3분 소요)
2. 상태가 **실행 중**으로 바뀌면 접속:
   - 강사 PC: `http://localhost:3000`
   - 학생(같은 와이파이): 상단에 표시된 `학생 접속` 주소를 칠판/화면에 공유
3. 끝나면 **[■ 종료]** 버튼으로 정리.

## 1. 워밍업 — 둘러보기 (10분)

- 상품을 구경하고, 장바구니에 담아보고, 회원가입을 해본다.
- **개발자 도구(F12)** 를 열어 Network 탭에서 어떤 요청이 오가는지 관찰한다.
- 질문: "내가 입력한 정보는 어디로 갈까? 누가 볼 수 있을까?"

## 2. 실습 A — SQL 인젝션으로 로그인 우회 (20분)

로그인 창에서 이메일 칸에 아래를 입력해 본다(비밀번호는 아무거나):

```
' OR 1=1 --
```

- 비밀번호를 모르는데 로그인이 되는 이유를 함께 토론한다.
- **핵심 개념:** 입력값이 SQL 쿼리에 그대로 끼어들어가면(인젝션) 인증을 우회당할 수 있다.
- **방어:** Prepared Statement / 파라미터 바인딩, 입력값 검증.

## 3. 실습 B — XSS 맛보기 (15분)

검색창이나 입력 폼에 다음을 넣어본다:

```
<iframe src="javascript:alert('XSS')">
```

- 경고창이 뜨면, 공격자가 임의의 스크립트를 남의 화면에서 실행시킬 수 있다는 뜻.
- **핵심 개념:** 사용자 입력을 그대로 화면에 출력하면 스크립트가 실행될 수 있다.
- **방어:** 출력 인코딩(이스케이프), Content-Security-Policy.

## 4. 마무리 토론 (15분)

- 오늘 본 취약점들의 공통 원인은? → "입력값을 믿었다"
- 개발자는 어떻게 막아야 할까?
- 이런 취약점을 **허락 없이 실제 사이트에 시도하면 범죄**라는 점을 강조.

---

### 강사 메모
- 더 많은 챌린지: Juice Shop 우측 상단 `Score Board`(`#/score-board`)에서 난이도별 과제 확인.
- 학생들이 동시에 접속해도 각자 브라우저 세션이라 충돌 없음.
- 데이터가 꼬이면 **종료 → 시작**으로 깨끗한 상태로 초기화됨.
